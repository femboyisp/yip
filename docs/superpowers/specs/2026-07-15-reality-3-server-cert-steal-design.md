# REALITY.3 — server: on-the-fly stolen-cert authed path + anti-replay + splice-faithful fallback — design spec

**Date:** 2026-07-15
**Status:** design — rev 3 (rev 2 folded the user's 8-point review; rev 3 folds an adversarial
advisor pass: TLS-1.3-only authed acceptor, per-SNI all-required startup, forged-cert staleness
bound, ring WINDOW+1 + monotonic clock, overflow metric, handshake/splice concurrency caps, explicit
threat-model boundary, and honest ServerHello-fidelity deferral to REALITY.5; corrected the earlier
passive-DPI overclaim). Pending user review.
**Parent:** [`2026-07-15-reality-tls-milestone-design.md`](2026-07-15-reality-tls-milestone-design.md)
**Depends on:** REALITY.1 (merged, the server front) + REALITY.2 (merged, the shared auth codec).
**Scope:** `yip-rendezvous` server only. PR 1 of the REALITY.3+.4 pair (REALITY.4 = client wiring, separate).

## Goal

Finalize the REALITY relay's **authenticated** branch so it is fully cover-faithful, and
close the two REALITY.1-review prerequisites that must land before a real client (REALITY.4)
connects:

1. **Stolen-cert authed TLS:** when an authenticated connection is accepted, terminate TLS
   presenting a certificate that mimics the real `dest` site (subject/SAN/validity/serial/keyUsage/
   EKU/basicConstraints/AIA), forged from `dest`'s live chain **pre-warmed at startup** — not the
   operator's own cert (no operator domain cert required, and the authed session's cert matches the
   borrowed identity). Matches Xray REALITY; enables optional REALITY-key-based outer-TLS server
   auth later.
2. **Anti-replay (REALITY.1 I-1):** reject a captured-and-replayed authed ClientHello at the
   pre-termination decision, via a sharded/time-bucketed/atomic dedup on the seal.
3. **Authed-but-inner-fail handling (REALITY.1):** an authed connection whose *inner* classification
   fails is **closed** — the branch is unreachable by replays (caught pre-termination by #2) or
   probers (no key), so the only reacher is a buggy own-peer; closing removes the non-splice-faithful
   decrypted-proxy exposure the review flagged.

## Threat model (boundary + what we do/don't claim)

An advisor review pushed on the central assumption, so state it explicitly:

- **Adversary WITHOUT config (the primary target).** No REALITY key ⇒ fails the seal ⇒ spliced to
  the real `dest` (REALITY.1). Byte-, cert-, and behavior-identical to the genuine site. This is the
  property REALITY exists to provide and the one we defend hardest.
- **Adversary WITH config (a legitimate client turned prober).** REALITY *cannot* hide from a party
  holding a valid key + short_id — they can auth, reach the authed acceptor, and observe that the
  forged cert doesn't chain to a public CA. This is REALITY's **accepted, documented limit** (same as
  Xray): possessing the config already tells them the IP runs a REALITY server. We do **not** claim to
  defend against this; we only ensure it reveals nothing *beyond* what the config already reveals, and
  we don't hand such a prober cheap extra distinguishers (hence the TLS-1.3-only pin in §1, the
  behavior-parity note in §3).
- **Passive DPI on an authed session — honest limitation (advisor #1).** The relay's *ServerHello*
  is generated by its own BoringSSL stack, not `dest`'s (nginx/Cloudflare/…). TLS 1.3 encrypts the
  *Certificate*, but the ServerHello (cipher choice, chosen group, extension set/order, GREASE
  placement) is cleartext and need not match `dest`'s template. Full server-side ServerHello mimicry
  is **out of scope for REALITY.3** and recorded as a follow-up (**REALITY.5: server-flight
  fidelity**). This spec does **not** claim passive-DPI indistinguishability for the authed path — it
  claims it only for the un-authed splice path (which is literally `dest`'s own bytes). Earlier wording
  ("passive DPI can't distinguish because the cert is encrypted") was an overclaim and is corrected.

## Background (current authed path)

`tls_front::run_reality_conn` (REALITY.1): reads the raw ClientHello, checks the REALITY seal.
**Authed** ⇒ `tokio_boring::accept(&acceptor, PrefixedStream::new(rec, tcp))` then
`conn::handle_connection`, where the acceptor is built by `build_acceptor(cert_path, key_path)`
from `--tls-cert`/`--tls-key`. `handle_connection` reads the first inner frame (obf-Register),
routing to the relay tunnel or, on classification failure, `into_decoy`.

## Design

### 1. Stolen-cert acceptor (`reality_cert.rs`, new module)

- **Pre-warm at STARTUP, never per-request (review #1).** Fetching `dest`'s cert on the first
  authed request would add a 50–500 ms latency spike to that handshake — a timing fingerprint. So
  the relay fetches `dest`'s chain **once at startup** (a `tokio_boring::connect` to `reality_dest`,
  SNI = each configured `reality_server_name`, bounded by a timeout), forges the acceptors, and
  caches them before serving traffic. A background task **refreshes** them (e.g. hourly) and, on a
  refresh failure, **keeps the last-good acceptor** — a transient `dest` outage never degrades the
  authed handshake.
- **No self-signed fallback for a real domain (review #2).** Serving a self-signed cert for
  `microsoft.com` on a fetch failure would instantly out the proxy. So there is **no runtime
  self-signed path** — ever. Startup granularity is **per-SNI, all-required** (advisor #6): the
  pre-warm must succeed for **every** configured server_name; if *any* one fails, the relay **fails
  to start** (not just "if all fail"). This closes the silent-hole case where a partial success would
  start a relay that has no acceptor for some authed SNI. (The operator fixes `dest`/network; a
  censor blocking egress at boot yields "relay won't start", which is the correct fail-closed outcome
  — better than leaking.)
- **TLS-1.3-only authed acceptor (advisor #2).** `build_acceptor` today uses
  `mozilla_intermediate_v5`, which **permits TLS 1.2** — where the Certificate message is sent in
  cleartext. A config-holding prober could then offer only TLS 1.2 and read the forged leaf
  (relay-ephemeral signature, no valid chain) with no decryption. The REALITY authed acceptor MUST
  therefore pin `set_min_proto_version(Some(SslVersion::TLS1_3))` (our own `yip-utls` client is
  already TLS-1.3-only, so this costs nothing). The legacy non-REALITY Trojan front keeps its
  existing profile.
- **Bounded staleness of the forged leaf (advisor #7).** The forged leaf copies `dest`'s
  serial/validity as of the last successful refresh; a config-holding prober comparing the authed
  (forged) serial against the un-authed (spliced, live) serial could spot drift on fast-rotating
  CDN/short-lived certs. Mitigation: refresh well inside the cert's own validity, and cap last-good
  staleness — if the last successful fetch for an SNI is older than `REALITY_CERT_MAX_STALE`
  (default 6 h), stop serving the authed path for that SNI and **splice** instead of presenting an
  ever-staler forgery. (Byte-perfect parity with a rotating live serial is unreachable and is noted,
  not claimed.)
- **Cache is bounded by the `reality_server_names` allowlist (review #7).** Certs are forged/cached
  ONLY for the configured server_names (a small, fixed set). An authed check already requires the
  SNI ∈ `server_names` (REALITY.1); an SNI outside it is un-authed ⇒ spliced to `dest`, and no cert
  is ever forged or cached for it. No SNI-flood OOM/CPU vector.
- **Forge a leaf, copying as much of `dest`'s cert as is reproducible (review #8).** Using `rcgen`,
  build a leaf copying `dest`'s leaf **subject, ALL subjectAltNames, notBefore/notAfter, serial,
  keyUsage, extendedKeyUsage, basicConstraints, and AIA (issuer/OCSP URLs)**, signed by a
  relay-ephemeral key generated at startup. **Honestly unreproducible:** the CA's signature and the
  embedded **SCTs** (signed by CT logs over the *original* cert/key) cannot be forged — they are
  bound to the real CA/log keys. This is acceptable and documented: in **TLS 1.3 the Certificate
  message is ENCRYPTED**, so these gaps are invisible to passive DPI; the mimicry serves the authed
  client (ours, zero-CA-auth) and any future REALITY-key-based client verification. Note the gap
  explicitly so we don't overclaim byte-perfect cert mimicry.
- **Serve it.** Build the authed-branch `SslAcceptor` per server_name from the forged `(cert, key)`
  (mozilla-intermediate profile, ALPN `h2`/`http/1.1`, matching `build_acceptor`) and select by the
  ClientHello SNI. `--tls-cert`/`--tls-key` become OPTIONAL when REALITY is configured (used only
  for the legacy non-REALITY Trojan front).

### 2. Anti-replay (`reality_replay.rs`, new type — sharded, time-bucketed, atomic)

**This layers UNDER an already-implemented stateless freshness gate — not instead of it (advisor
#3).** The seal's plaintext is `short_id (8) || ts_min (8 LE)`, and REALITY.1's `reality_auth_open`
already rejects any seal whose `ts_min` is outside ±`skew_min` of now — *statelessly*, no memory
needed. So a seal replayed after the window (or after a relay restart that clears memory) is already
rejected by the timestamp check ⇒ spliced to `dest`; it does **not** reach local termination. The
dedup set below only has to catch replays **within** the skew window. The one residual is a replay
within the *same* window across a relay restart (memory lost, timestamp still valid) — a ≤10-min
window per restart, accepted as a minor, documented residual (not the unbounded-in-time distinguisher
the review first supposed).

A **time-bounded dedup set** keyed on the 32-byte seal (`legacy_session_id`). On an authed decision,
if the seal was seen within the freshness window (±`REALITY_SKEW_MIN` = 10 min), treat the connection
as **un-authed** (⇒ splice to `dest`). The review flagged three concrete failure modes in the naive
`Arc<Mutex<HashMap>>` + evict-on-insert design; the type is built to close all three:

- **Sharded, not one global lock (review #4).** `N` shards (`N` = 16, a power of two), each an
  independent `Mutex` over its own bucket set. A connection hits exactly one shard, chosen by the
  low bits of the seal (the seal is a MAC output — uniformly distributed — so shards stay balanced).
  Lock contention drops ~16× vs. a single mutex, and the critical section is a couple of hash probes.
- **Time-bucketed eviction, O(1) not O(N) (review #5).** Each shard holds a tiny ring of
  per-minute buckets (`HashSet<[u8;32]>`) covering `[now-10min, now]`. The ring has **`WINDOW+1` =
  11 slots, not 10 (advisor #8)** — with only `WINDOW` slots, `minute % WINDOW` maps minute *m* and
  *m-10* to the same slot, which would either evict a still-live minute (accepting a replay) or read
  a stale one. Each slot carries its own minute-stamp; on access, a slot whose stamp is older than
  the window is `clear()`ed and re-stamped before use — O(1) amortized, never a full-map scan, no
  per-insert sweep. **Clock source:** bucket rotation uses a **monotonic** clock (`Instant`-based
  minute counter latched at startup), not wall-clock, so an NTP step backward cannot stall eviction
  or misjudge the window. (The `ts_min` *skew* check in `open()` still uses wall-clock, by necessity
  — it must agree with the client's clock — but that path is stateless and already bounded.)
- **Atomic check-and-insert under one lock (review #6).** `check_and_remember(seal) -> Verdict`
  takes the shard lock **once**, rotates that shard's expired buckets, tests membership across the
  live buckets, and inserts into the current bucket — all before releasing. Two racing replays of
  the same seal serialize on the shard lock; exactly one sees `Fresh`, the other `Replay`. There is
  no check-then-insert gap.

Bounded memory: at most (seals seen in a ~10-min window) live entries, hard-capped per shard —
if a shard's current bucket exceeds `MAX_BUCKET` = 65 536 (a defense-in-depth ceiling; worst case
16 shards × 11 buckets × 65 536 × 32 B ≈ 370 MB), further inserts into that minute are dropped and
the connection is treated as **un-authed/splice** (fail-safe: a flood degrades to "spliced to dest",
never to unbounded RAM or to accepting replays). **Note the availability edge (advisor #9):** only
*valid* seals ever insert (an invalid seal splices without inserting), so overflow cannot be forced
by junk — it happens only under genuine authed load, and its effect is that a real client is spliced
(its inner handshake then fails against the real `dest`). That is fail-safe for secrecy but a silent
availability cliff, so overflow **must** bump a `reality_replay_overflow_total` counter/metric and
log at warn, giving the operator a signal (rather than the current silent demotion). Lives on
`TlsFrontCfg` as `Arc<ReplayGuard>`; the shard lock is held only for the O(1) check+rotate+insert,
never across the handshake.

### 3. Authed-but-inner-fail handling (`conn.rs`) — close, don't proxy (review #3)

The original spec proposed, on inner-classification failure of an *authed* connection, to open a
fresh TLS session to `dest` and proxy the **decrypted** application stream both ways. The review
correctly rejected this: **it is not splice-faithful.** Once we've terminated TLS we hold an
`SslStream`; we cannot byte-replay the original ClientHello to `dest`. Proxying decrypted bytes to
a fresh `dest` session means the peer validated *our* (forged-for-cover) certificate but then reads
*genuine* `dest` application data underneath it — a cert/content provenance mismatch a prober could
detect, i.e. it manufactures exactly the tell we're trying to avoid.

**The fix is to not do it.** Reason about who can actually reach this branch:

- A **replay** never reaches it — §2's anti-replay runs at the auth *decision* (pre-termination),
  so a replayed authed ClientHello is spliced to `dest` before any TLS is terminated.
- An **active prober** never reaches it — a prober has no valid REALITY key, so it fails the seal
  check and is spliced to `dest` (REALITY.1), never terminated against our acceptor.
- The **only** party reaching authed-then-inner-fail is a peer that holds a valid REALITY key (it
  is *ours*) but sent a malformed/unclassifiable inner frame — a bug or version skew, not an
  adversary we owe cover-faithfulness to.

So on authed-but-inner-fail we **do not proxy**. This removes the exposure, deletes a whole fragile
code path, and needs no `dest` threading into `conn.rs`. Cover-faithfulness against replays/probers
is fully carried by §2 + REALITY.1's pre-termination splice, which *are* byte-faithful. (Documented
as an explicit decision, reversing the original spec's §3.)

**Behavior parity, not a bare RST (advisor #4).** The advisor noted that a *config-holding* prober
who deliberately sends a bad inner frame can compare this path against the other two personalities
(splice / tunnel) and against how `dest` fails. A real origin that gets garbage post-handshake
answers at the application layer, not with a silent TCP drop. Since this path is reached only by a
key-holder (per the reasoning above), full parity is low-value, but a bare RST is a gratuitous tell.
So instead of an abrupt close we send a **minimal generic HTTP response** (e.g. `HTTP/1.1 400 Bad
Request` / an `h2` `GOAWAY` when ALPN negotiated `h2`) followed by a TLS `close_notify`, matching
the shape of a real server rejecting a malformed request. This is best-effort parity for a
key-holding prober; we do **not** claim byte-parity with `dest`'s specific error page (recorded, not
overclaimed).

### 4. Bounded concurrency on handshake + splice (advisor #5)

The advisor flagged the one genuinely unbounded resource the earlier draft ignored: every inbound
connection costs an X25519 (seal check) and every un-authed/replay/overflow connection opens a
**fresh TCP+TLS connection to `dest`**. A flood of cheap junk ClientHellos then (a) burns relay CPU
on per-connection ECDH and (b) opens one upstream socket to `dest` per junk connection — exhausting
the relay's own fd/socket budget **and** turning the relay into a DoS amplifier toward `dest`, which
may get the relay's IP blocked and destroy the disguise. So REALITY.3 adds:

- A **global `tokio::sync::Semaphore`** bounding concurrent in-flight outer-TLS handshakes
  (`--reality-max-inflight-handshakes`, default e.g. 1024); a connection that can't get a permit
  promptly is dropped at the TCP layer (which is what an overloaded real server does too).
- A **second semaphore** bounding concurrent live splices to `dest`
  (`--reality-max-splices`, default e.g. 2048); over the cap, new un-authed connections are closed
  rather than dialing `dest` — capping both fd growth and amplification toward `dest`.

Both caps are on the REALITY.1 front (this hardens it); they are simple permit-guards, not queues,
so latency for legitimate load is unaffected. Per-source rate limiting is noted as a follow-up
(needs a source-IP table; out of scope for this PR).

## Config

- Reuse REALITY.1's `--reality-dest`/`--reality-private-key`/`--reality-short-id`/`--reality-server-name`.
- `--tls-cert`/`--tls-key` become optional when `--reality-dest` is set (the stolen cert supersedes
  them for the authed branch). Keep them required for the legacy `--listen-tcp`-only Trojan front
  (no REALITY).
- New (optional, defaulted): `--reality-cert-refresh-secs` (default 3600, cert-refresh cadence);
  `--reality-cert-max-stale-secs` (default 21600 / 6 h, `REALITY_CERT_MAX_STALE`, past which an SNI's
  authed path degrades to splice); `--reality-max-inflight-handshakes` (default 1024);
  `--reality-max-splices` (default 2048).

## Testing / adversary
- Unit (cert): stolen-cert forgery copies subject / all SANs / validity / serial / keyUsage / EKU /
  basicConstraints / AIA from a sample `dest` leaf; startup fails (does not serve self-signed) when
  the `dest` fetch fails for **any** configured server_name; the cache holds exactly one acceptor per
  configured server_name and forges none for an unlisted SNI; the acceptor **rejects a TLS-1.2-only
  ClientHello** (min-proto pin); an SNI past `REALITY_CERT_MAX_STALE` degrades to splice.
- Unit (anti-replay): `check_and_remember` returns `Fresh` once and `Replay` for a same-window
  repeat, `Fresh` again after the bucket ages out; **the WINDOW+1 ring does not evict a still-live
  minute** (off-by-one guard) and a monotonic-clock step-back doesn't accept a replay; two concurrent
  calls with the same seal yield exactly one `Fresh`; shard distribution over random seals is roughly
  even; a per-minute flood past `MAX_BUCKET` degrades to `Replay`/splice and bumps the overflow metric.
- Unit (concurrency): past the handshake/splice semaphore caps, new connections are shed (no
  unbounded `dest` dial-out).
- netns/integration: an authed connection (crafted with `yip_utls::auth::seal` + a `yip_utls`-style
  hello) terminates TLS and the presented cert's subject/SAN match the configured `dest`'s (against
  a local stand-in `dest` TLS server); a replayed authed ClientHello is spliced to `dest`
  (pre-termination); an authed connection that sends a bogus inner frame is **closed** (not decoy,
  not proxied).
- Keep the REALITY.1 active-probe property intact (un-authed ⇒ `dest`).

## Risks
- **Startup depends on `dest` reachability** — the pre-warm must succeed for **every** configured
  server_name or the relay refuses to start (deliberate: no self-signed leak, no silent per-SNI
  hole). Bounded by a connect timeout; operators ensure `dest` is reachable at boot.
- **Server-flight (ServerHello) fidelity is NOT addressed here** — the authed path's ServerHello is
  BoringSSL's, not `dest`'s, and is cleartext; a passive per-dest template could distinguish authed
  traffic. Explicitly deferred to **REALITY.5**; this spec claims passive indistinguishability only
  for the un-authed splice path.
- **The forged cert won't CA-validate, and SCTs/CA-signature are unreproducible** — intended
  (zero-CA-auth outer; TLS 1.3 + the TLS-1.3-only pin keep the Certificate encrypted). A
  config-holding prober can still observe the non-chaining cert — REALITY's accepted limit (see
  Threat model). A future REALITY.4 option can add REALITY-key-based client-side verification.
- **Forged-cert staleness vs a fast-rotating live `dest` cert** — bounded by `REALITY_CERT_MAX_STALE`
  (degrade to splice past it); byte-exact serial parity with a rotating live cert is unreachable.
- **Anti-replay is per-process** — a multi-relay deployment behind one dest would not share the
  dedup set; out of scope here (single-relay), noted for a future shared-state follow-up. In-window
  replay across a single relay's restart is an accepted ≤10-min residual (see §2).
- **Concurrency caps drop load past the ceiling** — a determined flood past
  `--reality-max-inflight-handshakes` / `--reality-max-splices` is shed (like any overloaded server);
  this bounds fd/CPU and amplification toward `dest` at the cost of serving fewer connections under
  attack. Per-source rate limiting is a follow-up.

## Non-goals
- Client wiring / `reality://` scheme (REALITY.4). REALITY-key-based client cert verification
  (optional REALITY.4 follow-up). Changing the un-authed splice (REALITY.1, unchanged).
  Cross-relay shared anti-replay state (single-process only here).

## Success criteria
1. Authed branch presents a cert whose subject/SAN/validity/serial/keyUsage/EKU/basicConstraints/AIA
   copy `dest`'s live leaf, pre-warmed at startup (no first-request fetch spike), no operator cert
   required; the acceptor is **TLS-1.3-only** (no cleartext-cert path); if *any* configured
   server_name fails to pre-warm the relay refuses to start (never self-signed for a real domain);
   an SNI whose cert goes staler than `REALITY_CERT_MAX_STALE` degrades to splice.
2. A replayed authed ClientHello is rejected: out-of-window by the existing stateless `ts_min` skew
   gate, in-window by a sharded/time-bucketed/atomic dedup that is O(1)-evicting (WINDOW+1 slots,
   monotonic clock), contention-sharded, race-free, memory-bounded, and metered on overflow.
3. An authed-but-inner-classification-fail connection is **not proxied** — it returns a minimal
   generic error + `close_notify` (no decrypted-stream proxy, no decoy). The review's exposure is
   removed by construction.
4. Handshake and splice concurrency are bounded (semaphores), capping relay fd/CPU and amplification
   toward `dest` under flood.
5. REALITY.1's un-authed→`dest` property and all existing reality tests stay green. `forbid-unsafe`,
   no `as`, clippy clean.
